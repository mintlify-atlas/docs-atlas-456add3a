Scorecard runs a set of automated checks against an open source repository to assess its security posture. Each check targets a specific heuristic—such as whether a project pins dependencies, requires code review, or signs its releases—and returns a score from 0 to 10. Together, these scores are combined into a weighted aggregate that reflects the overall risk level of the project.

The following table lists all checks that Scorecard runs by default, along with each check's risk level and GitLab support status. Select any check name to jump to its detailed documentation on the [reference page](/checks/reference).

<Note>
  GitLab support is indicated per check. **Supported** means the check works fully on GitLab. **Validating** means the check runs on GitLab but results are still being verified for accuracy. **Unsupported** means the check does not run on GitLab repositories.
</Note>

| [CI-Tests](/checks/reference#ci-tests) | Does the project run tests in CI before pull requests are merged? | Low | Supported |
| [CII-Best-Practices](/checks/reference#cii-best-practices) | Has the project earned an OpenSSF Best Practices Badge? | Low | Validating |

## Risk levels

Each check is assigned a risk level that reflects the potential impact of a low score. Risk levels are also used to weight individual check scores when computing the [aggregate score](/checks/scoring).

| Risk level | Weight | Example checks |
|---|---|---|
| Critical | 10 | Dangerous-Workflow, Webhooks |
| High | 7.5 | Binary-Artifacts, Branch-Protection, Code-Review, Maintained, Signed-Releases, Token-Permissions, Vulnerabilities |
| Medium | 5 | Fuzzing, Packaging, Pinned-Dependencies, SAST, Security-Policy |
| Low | 2.5 | CI-Tests, CII-Best-Practices, Contributors, License |
